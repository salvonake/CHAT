1) Executive Summary
=================================================================
LegalAI is a multi-host .NET 8 solution for evidence-constrained question answering
over private document corpora. The repository is organized into layered libraries
(Domain, Application, Ingestion, Retrieval, Security, Infrastructure) and four host
applications:

- LegalAI.Api (JWT-protected web API)
- LegalAI.WorkerService (Windows worker for file-based ingestion)
- LegalAI.Desktop (WPF desktop shell with fail-closed guard and tray lifecycle)
- LegalAI.Management.Api (fleet heartbeat and command control plane)

Compared with older repository reports, the current codebase is materially more
complete at host composition level: both API and Worker now include full DI wiring,
storage registration, model provider selection, and startup initialization services.

The system supports both local/offline paths (LLamaSharp + ONNX + embedded vectors)
and service-backed paths (Ollama + Qdrant), with scope-aware isolation via
domain/dataset/case namespace controls.

3) Architecture Layers and Dependency Flow
=================================================================
Layered structure (logical):

1. Domain Layer
- Core entities, interfaces, value objects, domain module contracts
- No host/framework-specific dependencies

2. Application Layer
- CQRS handlers via MediatR
- Ingestion and question-answer orchestration
- Uses domain abstractions, not infrastructure concretions

3. Specialized Processing Layers
- Ingestion: PDF extraction, chunking, embedding adapters
- Retrieval: query analysis, hybrid retrieval, context assembly
- Security: encryption and prompt-injection analysis

4. Infrastructure Layer
- Concrete implementations for:
  - Vector stores (Qdrant, embedded)
  - Data stores (SQLite doc/chat/identity/datasets/jobs)
  - LLM connectors (Ollama, LLamaSharp)
  - Telemetry and audit persistence

5. Host Layer
- API, worker service, desktop app, management API
- Composition roots bind abstractions to concrete providers

High-level dependency direction:
Hosts -> Application/Domain abstractions -> Infrastructure implementations

4) Host Applications and Runtime Responsibilities
=================================================================
4.1 LegalAI.Api (src/LegalAI.Api)
- ASP.NET Core Web API host
- JWT authentication with role/policy authorization
- Policies:
  - CanRead: Viewer, Analyst, Admin
  - CanQuery: Analyst, Admin
  - CanIngest: Analyst, Admin
- Registers retrieval, ingestion, stores, audit, metrics, prompt templates,
  injection detector, domain registry, and startup services
- Supports central management heartbeat publishing

Controller surface (primary):
- AskController
- AuthController
- ChatsController
- DatasetsController
- DocumentsController
- IngestionJobsController
- OpsController

4.2 LegalAI.WorkerService (src/LegalAI.WorkerService)
- .NET Worker with Windows service integration
- Service name: LegalAI WorkerService
- Watches configured PDF directory, queues events, ingests incrementally,
  retries with backoff, quarantines persistent failures
- Publishes heartbeat to management API and processes fleet commands

4.3 LegalAI.Desktop (src/LegalAI.Desktop)
- WPF WinExe host for local interactive operations
- Single-instance enforcement (mutex + named pipe activation)
- Fallback window activation if pipe signaling fails
- Tray lifecycle with minimize-to-tray and restore/focus behavior
- FailClosedGuard gates answer capability based on model/LLM/vector health
- Startup setup wizard when local model assets are missing

4.4 LegalAI.Management.Api (src/LegalAI.Management.Api)
- Minimal API for instance heartbeat and command routing
- In-memory state for instance status and command queues
- Optional API-key authorization headers:
  - X-Management-Key for operator actions
  - X-Agent-Key for service agents

6) Core Pipelines
=================================================================
6.1 Ask Pipeline
Primary handler: AskLegalQuestionHandler

Execution stages:
1. Prompt injection analysis and optional block
2. Query normalization + variant generation
3. Hybrid retrieval via vector + optional lexical fusion
4. Similarity-based abstention gate
5. Evidence-constrained generation prompt assembly
6. LLM generation
7. Citation extraction and confidence scoring
8. Optional strict-mode validation and regeneration
9. Metrics + audit logging

6.2 Retrieval Pipeline
Primary class: LegalRetrievalPipeline

Stages:
1. Query analysis
2. Variant embedding and vector search
3. Optional BM25 score fusion
4. Deduplication by content hash
5. Top-K selection
6. Context-budget packing with citation headers
7. Short-term result cache

6.3 Ingestion Pipeline
Primary handler: IngestDocumentHandler

Stages:
1. File hash and duplicate short-circuit
2. Upsert indexing state record
3. PDF extraction
4. Domain-aware chunking
5. Batched embedding generation
6. Vector upsert
7. Document status finalization
8. Retry/failure accounting and quarantine
9. Metrics + audit events

6.4 File Watcher Ingestion
Worker:
- Watches recursively for PDF changes
- Debounces events
- Priority queue processing in throttled batches
- Exponential retry and quarantine workflow

Desktop:
- Similar watcher pattern for local interactive mode
- Raises UI events for progress and ingestion status

=================================================================

  must be overridden in real deployments.

8) Desktop Reliability and Visibility Model
=================================================================
Single-instance model:
- Mutex: Local\\LegalAI_LCDSS_SingleInstance
- Activation named pipe: LegalAI_LCDSS_Activation

Secondary launch behavior:
1. Attempt named-pipe ACTIVATE message
2. Fallback to existing window handle activation
3. If both fail, user-facing background-running message

Visibility lifecycle:
- Minimize/close to tray configurable via appsettings
- Tray icon context menu for reopen/exit
- Foreground fallback notification when focus steal is blocked
- Off-screen recovery logic to reposition window on visible displays

Fail-closed behavior:
- Guard checks model presence/integrity, LLM availability, vector health
- Mode switches to library-only when blocking checks fail
